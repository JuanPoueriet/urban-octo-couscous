================================================================================
  1. ANALISIS DEL FUNCIONAMIENTO
================================================================================

1.1 ARQUITECTURA GENERAL
--------------------------

El menu movil esta implementado como un "drawer" lateral deslizante, situado
a la izquierda de la pantalla (o a la derecha en layouts RTL). Su arquitectura
es modular y desacoplada:

  MobileMenu (componente raiz)
    DrawerTransitionCoordinator  (maquina de estados FSM)
    MobileMenuGestures           (reconocimiento de gestos)
    MobileMenuAccessibility      (gestion de foco/ARIA)
    MobileMenuSearchController   (filtrado de secciones)
    MobileMenuSection            (acordeon de seccion)
    MobileMenuQuickAccess        (accesos directos)
    MobileMenuSearch             (campo de busqueda)

  Servicios globales:
    GestureBusService     (arbitraje global de puntero)
    MenuService           (senal de estado abierto/cerrado)
    OverlayManagerService (gestion de inert/background)
    ScrollLockService     (bloqueo de scroll)
    BreakpointService     (deteccion de breakpoint movil)

El estado global del menu es una senal Angular de tipo boolean
(MenuService.isMobileMenuOpen) que fluye hacia un efecto reactivo en
MobileMenu, el cual orquesta la FSM del drawer.

1.2 MAQUINA DE ESTADOS DEL DRAWER (FSM)
-----------------------------------------

El drawer maneja 5 estados en DrawerState:

  CLOSED -- abrir --> OPENING -- fin animacion --> OPEN
     ^                                               |
     |                                               | cerrar
     |                                               v
  CLOSED <-- fin animacion -- CLOSING <-- cerrar -- OPEN
                                  ^
                                  |
                            DRAGGING <-- gesto -- (cualquier estado)
                                  |
                                  +-- soltar --> OPENING o CLOSING

La transicion DRAGGING -> {OPENING, CLOSING} depende de velocidad, progreso y
distancia del gesto al soltarse. La FSM se implementa en
DrawerTransitionCoordinator y esta bien protegida contra transiciones ilegales
(guards de estado al inicio de openDrawer/closeDrawer).

1.3 SISTEMA DE GESTOS
----------------------

MobileMenuGestures implementa la interfaz GestureHandler y se registra en
GestureBusService, que escucha eventos de puntero globales (pointerdown,
pointermove, pointerup, pointercancel) con capture:true.

El sistema distingue tres tipos de gesto:

  A) EDGE SWIPE (apertura desde borde)
     - Se activa cuando el menu esta cerrado Y la pantalla es movil.
     - El umbral de borde es GESTURE_EDGE_THRESHOLD = 30px (adaptable a
       viewports < 360px con factor 1.2).
     - Al tocar el borde izquierdo (LTR) y arrastrar hacia la derecha, el
       drawer sigue el dedo con resistencia elastica al sobrepasar el limite.
     - Al soltar: si velocity > 0.25 px/ms, o progreso > 30%, o distancia
       > 30px, el drawer se abre completamente; en caso contrario, rebota
       a cerrado.

  B) DRAWER DRAG (cierre desde el drawer abierto)
     - Se activa cuando el menu esta abierto o en animacion, y el toque
       cae dentro del area del drawer.
     - El gesto horizontal desplaza el drawer con elasticidad al sobrepasar
       el limite abierto (0px). Se captura la posicion visual actual para
       soportar interrupciones mid-animacion.
     - La direccion vertical bloquea el gesto horizontal (GESTURE_VERTICAL_
       LOCK_THRESHOLD = 10px), permitiendo scroll vertical normal.

  C) OVERLAY INTERACTION (cerrar desde el overlay)
     - Se activa cuando el menu esta abierto y el toque cae sobre el overlay.
     - Un tap (movimiento < tapThreshold adaptativo) cierra el menu.
     - Un swipe horizontal en direccion de cierre con velocidad o distancia
       suficiente tambien cierra el menu.

1.4 ANIMACIONES Y TRANSICIONES
---------------------------------

La animacion principal es un transform:translateX con:
  - Duracion: 400ms
  - Easing: cubic-bezier(0.16, 1, 0.3, 1) (spring-like)

El overlay usa una variable CSS --mm-overlay-progress (0 a 1) para calcular
opacidad mediante una curva cuadratica eased:
  --mm-overlay-eased = 1 - (1 - progress)^2
  opacity = 0.08 + eased * 0.62

Los acordeones de secciones usan CSS Grid (grid-template-rows: 0fr -> 1fr)
para animar la altura real del contenido, evitando el antipatron max-height.

La funcion handleTransitionEnd escucha el evento 'transitionend' con filtro
propertyName === 'transform', y tiene un timer de fallback de 500ms para
evitar que la FSM quede bloqueada si el evento no dispara (p.ej. si la
animacion es interrumpida o el elemento esta oculto).

Se respeta prefers-reduced-motion: la transicion se establece a 'none' y
el timer de fallback dispara inmediatamente (0ms).

1.5 ACCESIBILIDAD
------------------

El drawer usa role="dialog" y aria-modal="true". Cuando esta cerrado se aplica
el atributo 'inert', eliminandolo del arbol de accesibilidad.

OverlayManagerService aplica 'inert' a los elementos de fondo
('main, jsl-footer, jsl-top-bar, .header__nav') cuando hay overlays activos,
evitando que los lectores de pantalla accedan al contenido de fondo.

MobileMenuAccessibility gestiona:
  - Guardado y restauracion del foco al abrir/cerrar.
  - Trap de foco con Tab/Shift+Tab.
  - Focus sentinels al inicio y final del drawer para ciclar el foco.
  - Refresco reactivo de la lista de focusables tras cambios en el DOM
    (MutationObserver) y tras busquedas (afterNextRender).

Tecla Escape cierra el menu (en MobileMenu.handleKeyboardEvent).

El boton hamburguesa en el header tiene aria-expanded y aria-label.
Los acordeones tienen aria-expanded y aria-controls correctamente enlazados.

1.6 COMPORTAMIENTO DE BUSQUEDA
---------------------------------

El campo de busqueda filtra los enlaces del menu en tiempo real. El
MobileMenuSearchController usa una cache de traducciones (invalidada en
cambio de idioma) y auto-expande hasta 2 secciones con resultados. Al borrar
la busqueda, restaura el estado de expansion previo.

1.7 BOTON HAMBURGUESA (apertura manual)
-----------------------------------------

El boton .header__mobile-toggle en Header llama a menuService.toggle(),
que actualiza la senal. El efecto en MobileMenu reacciona y lanza
openDrawer() o closeDrawer() segun corresponda.

El boton cambia de icono (Menu -> X) y rota 90 grados con transicion CSS
cuando el menu esta abierto.

Cuando el viewport supera el breakpoint de 992px, el efecto en MobileMenu
cierra automaticamente el menu con razon 'system', evitando que el drawer
quede abierto en desktop.


================================================================================
  2. DETECCION DE PROBLEMAS
================================================================================

Los problemas se listan de mayor a menor severidad (ALTA / MEDIA / BAJA / INFO).

------------------------------------------------------------------------------
PROBLEMA 1 [SEVERIDAD: ALTA]
El overlay no se hace visible durante el gesto de edge-swipe (apertura)
------------------------------------------------------------------------------

Archivos afectados:
  mobile-menu.ts (lineas 341-344)
  drawer-transition-coordinator.ts (lineas 76-88)
  mobile-menu.scss (lineas 688-728)

Descripcion:
  Durante un gesto de edge-swipe para abrir el menu, el estado del drawer
  transiciona a DRAGGING. En ese estado, DrawerTransitionCoordinator solo
  llama a clearTransitionListeners() y NO anade la clase '.visible' al overlay.

  El overlay tiene 'visibility: hidden' por defecto y solo se hace visible
  cuando se anade '.visible' en la transicion a OPENING. Por tanto, aunque
  updateOverlayVisual(progress) actualice --mm-overlay-progress correctamente
  durante el drag, la propiedad 'visibility: hidden' impide que los pixeles
  se dibujen.

  El comentario en mobile-menu.scss (linea 719) afirma:
    "This approach also makes the overlay visible during edge-swipe gestures"
  Esta afirmacion es INCORRECTA. El overlay permanece invisible durante
  el edge-swipe.

  Consecuencia para el usuario:
  El drawer aparece deslizandose desde el borde sin ningun oscurecimiento
  progresivo del fondo. La transicion se percibe menos pulida y el feedback
  visual sobre la "intencion de apertura" se pierde.

Pasos para reproducir:
  1. En movil (< 992px), menu cerrado.
  2. Iniciar un gesto desde el borde izquierdo de la pantalla arrastrando
     hacia la derecha.
  3. Observar que el drawer se mueve pero el fondo NO se oscurece
     progresivamente.

------------------------------------------------------------------------------
PROBLEMA 2 [SEVERIDAD: MEDIA]
El gesto de deslizamiento sobre el overlay no produce feedback visual
------------------------------------------------------------------------------

Archivo afectado: mobile-menu-gestures.ts (lineas 331-356)

Descripcion:
  Cuando el menu esta abierto y el usuario desliza horizontalmente sobre el
  overlay (gesto de cierre), handleOverlayMove solo actualiza el buffer de
  velocidad pero NO mueve el drawer. El comentario en el codigo lo reconoce:
    "We could also update the drawer position here for a parallax feel.
     For now, just tracking the gesture for the end decision."

  El drawer permanece inmovil durante el swipe y solo actua al soltar el
  dedo (si se cumple el umbral de velocidad o distancia).

  Consecuencia para el usuario:
  El gesto de cierre sobre el overlay se siente "sordo": no hay feedback
  en tiempo real. El usuario puede creer que el gesto no esta siendo
  reconocido y puede repetirlo o interrumpirlo. Contrasta negativamente con
  el drag desde el drawer, que si tiene feedback visual sincronizado.

------------------------------------------------------------------------------
PROBLEMA 3 [SEVERIDAD: MEDIA]
CSS body.no-scroll tiene 'position' duplicado que anula la correccion de iOS
------------------------------------------------------------------------------

Archivo afectado: header.scss (lineas 342-348)

Descripcion:
  El bloque CSS para bloquear scroll es:

    body.no-scroll {
      overflow: hidden !important;
      position: fixed;     <- intencion: evitar bounce en iOS
      width: 100%;
      height: 100%;
      position: relative;  <- SOBREESCRIBE la linea anterior
    }

  La segunda declaracion 'position: relative' anula 'position: fixed'.
  En iOS Safari, position:fixed es necesario para evitar que el body haga
  scroll-bounce cuando hay un overlay. Con position:relative, el efecto
  de bloqueo en iOS no funciona correctamente y el usuario puede hacer
  scroll del fondo mientras el menu esta abierto.

  Nota: ScrollLockService puede implementar su propio mecanismo independiente.
  Si no usa esta clase CSS, el problema no tiene impacto funcional pero es
  codigo muerto y enganoso.

------------------------------------------------------------------------------
PROBLEMA 4 [SEVERIDAD: MEDIA]
Los acordeones colapsados no tienen aria-hidden, exponiendolos a lectores
de pantalla
------------------------------------------------------------------------------

Archivo afectado: mobile-menu-section.ts (lineas 28-61)

Descripcion:
  Las secciones del menu usan CSS Grid (grid-template-rows: 0fr) para
  colapsar visualmente el contenido cuando isExpanded = false. Sin embargo,
  el div con role="region" no recibe aria-hidden="true" ni el atributo
  'inert' cuando esta colapsado.

  Un lector de pantalla puede acceder a los enlaces de todas las secciones
  colapsadas aunque el usuario no las haya expandido, duplicando la
  informacion audible y desorientando al usuario. El aria-expanded en el
  boton del header del acordeon indica el estado pero no oculta el contenido.

------------------------------------------------------------------------------
PROBLEMA 5 [SEVERIDAD: MEDIA]
GestureBusService nunca elimina sus event listeners del documento
------------------------------------------------------------------------------

Archivo afectado: gesture-bus.service.ts (lineas 27-35)

Descripcion:
  El servicio registra en su constructor cuatro listeners de puntero
  sobre 'document' con capture:true (pointerdown, pointermove, pointerup,
  pointercancel) pero no existe ningun metodo de limpieza.

  Para un servicio providedIn:'root' que vive toda la vida de la aplicacion
  esto no produce memory leaks en produccion, pero:
    - En tests, cada spec que instancie el servicio anade nuevos listeners
      globales sin limpiarlos, causando acumulacion entre tests y
      comportamientos no deterministicos.
    - Si en el futuro el servicio se scope-a a un modulo lazy, la destruccion
      del modulo dejara listeners huerfanos sobre el document.

------------------------------------------------------------------------------
PROBLEMA 6 [SEVERIDAD: BAJA]
refreshFocusableElements fuerza multiples reflows (getBoundingClientRect)
------------------------------------------------------------------------------

Archivo afectado: mobile-menu-accessibility.ts (lineas 75-93)

Descripcion:
  executeRefresh() llama a el.getBoundingClientRect() por cada elemento
  focusable encontrado dentro del loop de filter:

    this.focusableElements = allPotential.filter(el => {
      const rect = el.getBoundingClientRect(); // fuerza reflow
      return ... rect.width > 0 && rect.height > 0;
    });

  Con 5 secciones x hasta 9 enlaces + botones y tiles, el menu puede
  tener 50+ elementos focusables. Cada getBoundingClientRect() en un loop
  sin batching fuerza un reflow del layout. Este metodo se llama:
    - En cada cambio en el MutationObserver (acordeones, busquedas).
    - En cada keystroke del campo de busqueda (via afterNextRender).

  En dispositivos de gama baja, esto puede provocar jank perceptible durante
  la busqueda interactiva.

------------------------------------------------------------------------------
PROBLEMA 7 [SEVERIDAD: BAJA]
OverlayManagerService.rootContentSelector esta hardcodeado
------------------------------------------------------------------------------

Archivo afectado: overlay-manager.service.ts (linea 17)

Descripcion:
  private readonly rootContentSelector = 'main, jsl-footer, jsl-top-bar, .header__nav';

  Si se anaden nuevas secciones al layout (nueva barra lateral, footer
  secundario, widget flotante...), el selector debe actualizarse manualmente
  en este servicio. No hay ningun mecanismo de registro dinamico. Un cambio
  de nombre de componente o clase CSS rompera silenciosamente el comportamiento
  inert del fondo.

------------------------------------------------------------------------------
PROBLEMA 8 [SEVERIDAD: BAJA]
onMenuPointerMove, onMenuPointerEnd y onMenuPointerCancel son publicos
innecesariamente
------------------------------------------------------------------------------

Archivo afectado: mobile-menu-gestures.ts (lineas 358, 458, 513)

Descripcion:
  Estos tres metodos son publicos pero son llamados exclusivamente desde
  metodos privados de la misma clase:

    private handleMenuDragMove   = (e) => this.onMenuPointerMove(e);
    private handleMenuDragEnd    = (e) => this.onMenuPointerEnd(e);
    private handleMenuDragCancel = (e) => this.onMenuPointerCancel(e);

  Al ser publicos, cualquier codigo externo puede invocarlos con eventos
  arbitrarios, corrompiendo el estado de la FSM (activePointerId, isDragging,
  velocityBuffer). Violan el principio de minima superficie publica.

------------------------------------------------------------------------------
PROBLEMA 9 [SEVERIDAD: BAJA]
ViewEncapsulation.None puede producir colisiones de clases CSS
------------------------------------------------------------------------------

Archivos afectados: mobile-menu.ts (linea 76), mobile-menu-section.ts (linea 9),
                    mobile-menu-quick-access.ts (linea 8)

Descripcion:
  Todos los componentes del menu usan ViewEncapsulation.None, lo que hace
  que sus estilos sean globales. Clases como .mobile-quick-access,
  .mobile-accordion, .mobile-links-list, .mobile-menu-content no tienen
  prefijo de encapsulacion.

  Cualquier otro componente del proyecto que use nombres identicos sufrira
  colisiones de estilos. A medida que el proyecto escale y se anadan mas
  features, el riesgo de colision aumenta. Para componentes de UI complejos,
  encapsulation:None es aceptable solo si las clases CSS son suficientemente
  unicas (p.ej. con prefijo 'jsl-mobile-menu__').

------------------------------------------------------------------------------
PROBLEMA 10 [SEVERIDAD: INFO]
Doble declaracion de opacity en .mobile-menu-overlay (la primera es letra muerta)
------------------------------------------------------------------------------

Archivo afectado: mobile-menu.scss (lineas 693-702)

Descripcion:
  .mobile-menu-overlay {
    opacity: 0;     <- declaracion muerta, sobreescrita inmediatamente
    ...
    opacity: calc(0.08 + var(--mm-overlay-eased) * 0.62);  <- efectiva
  }

  La primera declaracion 'opacity: 0' nunca tiene efecto porque la segunda
  la sobreescribe en la misma regla CSS. Es confuso para futuros desarrolladores
  que pueden asumir que la opacidad base es 0 cuando en realidad es ~0.08.

------------------------------------------------------------------------------
PROBLEMA 11 [SEVERIDAD: INFO]
GestureBusService no limpia activeHandler al inicio de un nuevo pointerdown
------------------------------------------------------------------------------

Archivo afectado: gesture-bus.service.ts (lineas 49-78)

Descripcion:
  En handlePointerDown, si ningun handler captura el evento, activeHandler
  permanece con su valor anterior (que deberia ser null tras el ultimo
  pointerup). Esto es un edge case menor porque activeHandler se resetea
  en handlePointerUp y handlePointerCancel, y los handlers individuales
  verifican activePointerId. Sin embargo, anadir this.activeHandler = null
  al inicio de handlePointerDown haria el codigo mas defensivo ante
  situaciones donde se pierda el evento pointerup (p.ej. pointer saliendo
  de la ventana del navegador).

================================================================================
  4. PROPUESTAS DE SOLUCION
================================================================================

------------------------------------------------------------------------------
SOLUCION AL PROBLEMA 1: Mostrar el overlay durante el edge-swipe
------------------------------------------------------------------------------

Por que es necesaria:
  El overlay actua como feedback visual de progreso. Sin el, el usuario ve
  el drawer aparecer desde el borde sin que el fondo cambie, lo que resulta
  en una animacion que se percibe incompleta o rota.

Pasos para corregirlo:

  Opcion A (minima invasiva): En DrawerTransitionCoordinator.transitionTo,
  caso DRAGGING, anadir logica para revelar el overlay si el estado anterior
  era CLOSED (gesto de apertura desde borde cerrado):

    case DrawerState.DRAGGING: {
      this.clearTransitionListeners();
      // Si veniamos de CLOSED, el gesto es de apertura: revelar overlay
      const wasClosed = (this.currentState === DrawerState.CLOSED);
      if (wasClosed) {
        const overlay = this.config.getOverlayElement();
        if (overlay) {
          overlay.classList.add('visible');
        }
      }
      break;
    }

  IMPORTANTE: guardar el estado previo antes de la asignacion this.currentState = newState.

  Opcion B (mas elegante): Redisenar la visibilidad del overlay para que
  dependa exclusivamente del CSS variable --mm-overlay-progress, eliminando
  la dependencia de la clase '.visible' para la renderizacion:

    En mobile-menu.scss:
      .mobile-menu-overlay {
        /* Ya no usar visibility: hidden en el estado base */
        opacity: calc(0.08 + var(--mm-overlay-eased, 0) * 0.62);
        pointer-events: none;  /* mantener; se cambia con .visible */
        visibility: visible;   /* siempre visible; solo opacity cambia */
        ...

        &.visible {
          pointer-events: auto;
          /* visibility ya es visible en el base state */
        }
      }

  Con la Opcion B, el overlay siempre renderiza y su opacidad viene dictada
  por --mm-overlay-progress. Cuando el menu esta cerrado y no hay drag,
  --mm-overlay-progress = 0, por lo que opacity = 0.08 (practicamente
  invisible). No es necesario tocar la clase .visible para la visibilidad.

  Nota: Verificar que con la Opcion B el overlay no interfiere con clicks
  sobre el contenido de la pagina cuando el menu esta cerrado. El
  pointer-events:none en el estado base lo garantiza.

------------------------------------------------------------------------------
SOLUCION AL PROBLEMA 2: Feedback visual al deslizar sobre el overlay
------------------------------------------------------------------------------

Por que es necesaria:
  En iOS/Android, el usuario espera que el elemento que esta arrastrando
  siga su dedo. Un overlay que no reacciona al swipe rompe la expectativa
  de "fisica directa" y crea la sensacion de latencia o fallo.

Pasos para corregirlo:
  En mobile-menu-gestures.ts, metodo handleOverlayMove, actualizar la
  posicion del drawer en tiempo real cuando se detecta un gesto horizontal:

    private handleOverlayMove(event: PointerEvent): void {
      this.currentX = event.clientX;
      this.currentY = event.clientY;
      const diffX = this.currentX - this.startX;
      const diffY = this.currentY - this.startY;
      const absDiffX = Math.abs(diffX);
      const absDiffY = Math.abs(diffY);

      if (!this.isDragging && !this.isHorizontalGesture) {
        if (absDiffY > this.verticalLockThreshold && absDiffY > absDiffX) return;
        if (absDiffX > this.horizontalThreshold && absDiffX > absDiffY) {
          this.isHorizontalGesture = true;
          this.isDragging = true;
          this.config.onToggleHaptic();
        }
      }

      if (this.isDragging && this.isHorizontalGesture) {
        this.pushVelocitySample(event.clientX);

        // NUEVO: mover el drawer con el dedo (parallax de cierre)
        const menuWidth = this.config.menuWidth;
        const isRtl = this.config.isRtl();
        const closedPos = this.getClosedPosition(menuWidth);
        const openPos = 0;
        // Limitar al rango [closedPos, openPos] en LTR o [openPos, closedPos] en RTL
        const targetTranslate = isRtl
          ? Math.min(closedPos, Math.max(openPos, openPos + diffX))
          : Math.max(closedPos, Math.min(openPos, openPos + diffX));
        const progress = this.getProgress(targetTranslate, menuWidth);
        this.lastDragPosition = targetTranslate;
        this.config.onUpdateTranslate(targetTranslate, progress, 1,
          isRtl ? 'right' : 'left');
      }
    }

  Esto garantiza que arrastrar sobre el overlay mueve el drawer en tiempo
  real, haciendo la experiencia equivalente al drag directo sobre el drawer.
  La logica de decision al soltar (handleOverlayEnd) permanece igual.

SOLUCION AL PROBLEMA 3: Corregir body.no-scroll en header.scss
------------------------------------------------------------------------------

Por que es necesaria:
  Si ScrollLockService usa esta clase para bloquear el scroll en iOS, la
  doble declaracion hace que position:fixed no tenga efecto, permitiendo
  scroll-bounce del fondo en iOS Safari mientras el menu esta abierto.

Pasos para corregirlo:
  En header.scss, lineas 342-348, eliminar la segunda declaracion position:

    body.no-scroll {
      overflow: hidden !important;
      position: fixed;     /* MANTENER SOLO ESTA */
      width: 100%;
      height: 100%;
      /* ELIMINAR: position: relative; */
    }

  Accion adicional: Verificar si ScrollLockService usa esta clase CSS. Si
  implementa su propio mecanismo independiente, evaluar si el bloque
  body.no-scroll puede eliminarse completamente de header.scss para evitar
  codigo muerto.

------------------------------------------------------------------------------
SOLUCION AL PROBLEMA 4: aria-hidden en acordeones colapsados
------------------------------------------------------------------------------

Por que es necesaria:
  Sin aria-hidden, los lectores de pantalla leen todos los enlaces de todas
  las secciones aunque esten visualmente colapsadas. Con 5 secciones y hasta
  9 enlaces cada una, esto crea una lista de ~45 elementos navegables no
  contextualizados.

Pasos para corregirlo:
  En mobile-menu-section.ts, anadir [attr.aria-hidden] al div del contenido:

    <div
      [id]="'accordion-' + sectionId + '-content'"
      class="accordion-content"
      role="region"
      [attr.aria-labelledby]="'accordion-' + sectionId + '-header'"
      [attr.aria-hidden]="!isExpanded ? 'true' : null"
    >

  Esto ocultara el contenido colapsado de los lectores de pantalla.

  Alternativa mas robusta: usar el atributo 'inert' en lugar de aria-hidden
  para tambien evitar el foco con Tab en el contenido colapsado:

    [attr.inert]="!isExpanded ? '' : null"

  Con 'inert', los enlaces dentro del acordeon colapsado tambien se excluyen
  de la navegacion por teclado, lo que es coherente con su estado visual.

------------------------------------------------------------------------------
SOLUCION AL PROBLEMA 5: Limpiar event listeners de GestureBusService
------------------------------------------------------------------------------

Por que es necesaria:
  Previene acumulacion de listeners en entornos de testing y hace el servicio
  mas robusto ante cambios de scope en el futuro.

Pasos para corregirlo:
  Implementar ngOnDestroy en GestureBusService (requiere que Angular lo
  gestione como un servicio con ciclo de vida):

    ngOnDestroy(): void {
      document.removeEventListener('pointerdown',   this.handlePointerDown,   { capture: true });
      document.removeEventListener('pointermove',   this.handlePointerMove,   { capture: true });
      document.removeEventListener('pointerup',     this.handlePointerUp,     { capture: true });
      document.removeEventListener('pointercancel', this.handlePointerCancel, { capture: true });
    }

  Los event listeners son funciones de flecha asignadas como propiedades del
  servicio, por lo que removeEventListener puede eliminarlas correctamente
  (misma referencia de funcion).

  Nota: Los servicios providedIn:'root' en Angular no tienen ngOnDestroy
  invocado automaticamente. Para garantizar la limpieza, se puede exponer
  un metodo destroy() publico llamado desde los tests, o usar un patron
  DestroyRef/takeUntilDestroyed si se refactoriza.

------------------------------------------------------------------------------
SOLUCION AL PROBLEMA 6: Eliminar forced reflows en refreshFocusableElements
------------------------------------------------------------------------------

Por que es necesaria:
  getBoundingClientRect() en un loop sin batching provoca un reflow por cada
  llamada. Para menus con 50+ elementos en dispositivos de gama baja, esto
  puede causar jank durante la busqueda interactiva.

Pasos para corregirlo:
  Sustituir la verificacion por getBoundingClientRect por una verificacion
  mediante computedStyle, que no fuerza reflow individual:

    private executeRefresh() {
      const focusableSelectors =
        'a[href], button:not([disabled]), textarea, input, select, [tabindex]:not([tabindex="-1"])';

      const allPotential = Array.from(
        this.el.nativeElement.querySelectorAll(focusableSelectors)
      ) as HTMLElement[];

      this.focusableElements = allPotential.filter(el => {
        const style = window.getComputedStyle(el);
        return style.display !== 'none' &&
               style.visibility !== 'hidden' &&
               parseFloat(style.opacity) > 0 &&
               !el.hasAttribute('inert') &&
               !el.closest('[inert]');
      });
    }

  getComputedStyle no fuerza reflow cuando se llama en un loop (el motor
  puede batching las consultas CSS). Solo si se necesita verificar elementos
  con size explicito a cero (casos extremadamente raros en este menu) se
  justificaria getBoundingClientRect.

SOLUCION AL PROBLEMA 7: Hacer rootContentSelector configurable
------------------------------------------------------------------------------

Por que es necesaria:
  Un selector hardcodeado crea acoplamiento implicito entre el servicio y la
  estructura del layout. Si se renombra un componente, el comportamiento inert
  se rompe silenciosamente sin ningun error en compilacion o runtime.

Pasos para corregirlo:
  Implementar un mecanismo de registro dinamico en OverlayManagerService:

    @Injectable({ providedIn: 'root' })
    export class OverlayManagerService {
      private inertTargets: Set<Element> = new Set();

      registerInertTarget(el: Element): () => void {
        this.inertTargets.add(el);
        this.updateInertState(); // actualizar estado inmediatamente si hay overlay activo
        return () => {
          this.inertTargets.delete(el);
        };
      }

      private updateInertState(): void {
        const hasActiveOverlays = this.overlayStack.length > 0;
        this.inertTargets.forEach((target) => {
          if (hasActiveOverlays) {
            target.setAttribute('inert', '');
          } else {
            target.removeAttribute('inert');
          }
        });
      }
    }

  Cada componente de layout (AppComponent, Footer, TopBar, Header nav)
  llamaria a registerInertTarget(this.el.nativeElement) en ngAfterViewInit
  y llamaria al unregister devuelto en ngOnDestroy. Esto elimina el
  acoplamiento implicito y hace el sistema extensible sin modificar el servicio.

FORTALEZAS:
  + Arquitectura muy modular (FSM + bus de gestos + a11y en clases separadas).
  + Soporte robusto de interrupciones: gestos mid-animacion, cancelaciones de
    sistema (llamada entrante), y resize de breakpoint.
  + Thresholds adaptativos por DPI y tamano de viewport.
  + Elasticidad configurable (rubber-band) en ambos extremos del drag.
  + Soporte completo de prefers-reduced-motion.
  + Focus trap, aria-modal e inert del fondo bien implementados.
  + Feedback haptico con fallback visual.
  + Busqueda en tiempo real con cache de traducciones e i18n.
  + RTL contemplado en toda la logica de gestos y posicionamiento.
  + Cleanup de recursos en ngOnDestroy (ResizeObserver, touchmove, gestos).

DEBILIDADES PRINCIPALES:
  - Overlay invisible durante edge-swipe (bug funcional visible, Problema 1).
  - Sin feedback visual al swipe sobre el overlay (UX degradada, Problema 2).
  - body.no-scroll CSS roto en iOS Safari (Problema 3).
  - Acordeones colapsados accesibles para screen readers (Problema 4).
